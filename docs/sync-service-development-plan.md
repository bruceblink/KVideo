# KVideo 付费同步服务开发计划

> 当前状态：商业化开发暂缓。现阶段只处理已有同步能力中的安全漏洞、数据丢失和严重兼容问题，不启动套餐扩展、设备收费、备份收费或支付接入。恢复条件和当前主线以 [播放器优先开发计划](player-first-development-plan.md) 为准。

## 1. 计划目标

KVideo 将保持视频聚合与播放客户端定位，在不限制本地核心能力的前提下，通过 SyncHub 提供可选的付费云同步服务。

付费服务只覆盖账户、跨设备同步、云端备份、数据恢复、设备管理和相关云端基础设施。KVideo 与 SyncHub 不销售、分发、提供或授权视频、IPTV、弹幕及其他内容资源，也不把视频源作为套餐权益。

本计划覆盖 `F:\project\KVideo` 与 `F:\project\SyncHub` 两个仓库。两者独立版本、测试、提交和发布，跨仓库功能只有在双方契约均验证通过后才能标记完成。

本计划是同步服务商业化启动后的候选路线，不是当前迭代队列。播放器质量和用户需求未达到启动门槛前，不应因为本计划章节完整而提前实现后续阶段。

## 2. 术语与边界

| 术语 | 明确定义 | 不代表什么 |
| --- | --- | --- |
| KVideo 账户 | 当前 KVideo 部署内用于登录、权限和本地数据隔离的账户 | SyncHub 订阅或付费身份 |
| SyncHub 账户 | 持有订阅、应用 API Key 和云端同步数据的独立账户 | 视频内容账户 |
| 角色 `role` | `viewer`、`admin`、`super_admin`，用于 KVideo 后台管理授权 | 套餐等级或 VIP 等级 |
| 套餐 `plan` | SyncHub 云服务的商业规格，例如 `free`、`pro` | KVideo 管理权限或内容权限 |
| 权益 `entitlement` | 服务端计算出的同步、备份、设备管理等具体能力 | 页面中可自行伪造的布尔值 |
| 来源配置 | 用户自行提供的视频、IPTV、弹幕或订阅源配置 | KVideo 或 SyncHub 提供的内容 |
| 来源订阅 | 定期拉取用户指定配置地址的技术功能 | 付费套餐或自动续费订阅 |
| 独立数据空间 | 当前代码中使用 `premium` 和 `/premium` 命名的第二套本地配置、收藏和历史空间 | VIP 内容区、付费视频区或高级来源目录 |
| 同步文档 | SyncHub 按用户、应用和 collection 保存的版本化 JSON 数据 | 视频文件或媒体资源 |
| 设备 | 一个安装或浏览器配置实例，使用稳定的客户端生成 ID 标识 | API Key 本身 |

### 2.1 永久产品边界

1. Free 用户可以在本地使用搜索、播放、收藏、历史和自行配置来源。
2. 套餐不得按视频源、搜索结果、清晰度或具体内容划分。
3. SyncHub 不维护、推荐或下发付费来源目录。
4. 来源可用性、内容授权和访问合法性由配置来源的部署者或用户负责。
5. 付费校验必须发生在 SyncHub 服务端，客户端展示不能作为授权依据。
6. `role` 与 `plan` 永远独立；管理员不会自动获得付费套餐，付费用户也不会获得后台权限。

## 3. 当前基线

### 3.1 KVideo 已有能力

- KVideo 托管账户、角色、细粒度管理权限和按 `profileId` 的数据隔离。
- SyncHub 服务地址和 KVideo 应用 API Key 配置、连接验证与撤销。
- `watch-history`、`favorites` 两个同步 collection。
- 浏览器直连 SyncHub 和 KVideo 服务端代理 SyncHub 两种接入方式。
- 无 SyncHub Key 时兼容现有 Redis 同步；本地能力不依赖 SyncHub。

### 3.2 SyncHub 已有能力

- PostgreSQL 用户、`free/pro` 订阅记录和计费周期字段。
- 应用限定 API Key，KVideo Key 不能访问其他应用的数据。
- 有效订阅校验、Key 撤销、最后使用时间和 1 MiB 单文档限制。
- 按用户、应用、collection 隔离的 JSON 文档与递增版本号。
- 账户页展示套餐、周期、续订状态和 API Key 管理。

### 3.3 必须补齐的问题

- SyncHub 当前只校验订阅状态，尚未把 `free/pro` 映射为明确权益和配额。
- KVideo 使用整份数组覆盖同步，多设备并发修改可能最后写入覆盖前一次修改。
- KVideo API Key 存在浏览器本地存储路径，缺少正式的设备身份、设备撤销和密钥轮换体验。
- 同步失败主要写入控制台，用户看不到最后成功时间、失败原因和恢复动作。
- 当前仅同步观看历史和收藏，没有设置、用户自有来源配置、备份历史和恢复能力。
- 代码中的 `premium` 容易被误解为商业套餐，需要独立迁移。

## 4. 产品规格

### 4.1 首发套餐

首发只保留 `Free` 与 `Pro`，避免在成本和需求尚未验证前引入家庭组或多档套餐。

| 能力 | Free | Pro |
| --- | --- | --- |
| KVideo 本地功能 | 完整 | 完整 |
| 云端观看历史与收藏同步 | 不提供 | 提供 |
| 设置同步 | 不提供 | 提供 |
| 用户自有来源配置同步 | 不提供 | 提供，客户端加密后上传 |
| 已登记设备数 | 0 | 初始上限 5 台，可配置 |
| 手动云端备份与恢复 | 不提供 | 提供 |
| 版本保留 | 不提供 | 初始 30 天，可配置 |

表中数值是 MVP 的运行参数，不是永久商业承诺。正式定价前需根据存储、请求量、支付渠道和支持成本复核。

### 4.2 首发权益标识

```text
metadata_sync
settings_sync
source_config_sync
cloud_backup
version_history
device_management
```

SyncHub 服务端根据套餐和订阅状态计算权益。KVideo 只能消费服务端结果，不在客户端自行推断套餐能力。

### 4.3 同步数据分类

| 数据 | 同步形式 | 合并规则 |
| --- | --- | --- |
| 观看历史 | 逐记录同步 | 同一记录按 `updatedAt` 合并，播放进度取较新修改 |
| 收藏 | 逐记录同步并保留删除标记 | 同一记录按 `updatedAt` 合并，删除参与冲突判断 |
| 普通设置 | 版本化文档 | 比较基础版本；冲突时保留两版并提示用户 |
| 用户自有来源配置 | 客户端加密的版本化文档 | 服务端只保存密文；冲突时不得静默覆盖 |
| 独立空间数据 | 暂不新增同步 | 等命名与数据模型迁移完成后单独决策 |

任何来源配置都不得包含在服务端日志、指标标签、错误追踪字段或管理页面明文预览中。

## 5. 系统职责

### 5.1 KVideo 负责

- 本地数据和离线可用性。
- 设备 ID 的生成、安全保存与设备名称采集。
- 同步队列、重试、幂等请求、冲突展示和同步状态 UI。
- 对敏感来源配置执行客户端加密和解密。
- 在无订阅、订阅到期或 SyncHub 不可用时继续提供本地能力。

### 5.2 SyncHub 负责

- SyncHub 账户认证、订阅状态、套餐权益和计费事件。
- KVideo 应用 API Key 的创建、散列存储、撤销和轮换。
- 设备登记、设备上限、最后活动时间和远程撤销。
- 同步文档版本、条件写入、冲突检测、备份快照和保留策略。
- 配额、限流、审计事件和不包含用户敏感正文的运行指标。

### 5.3 暂不合并的账户

MVP 不强制合并 KVideo 账户和 SyncHub 账户。一个 KVideo 部署中的用户通过自己的 KVideo 应用 API Key 绑定 SyncHub。未来若引入 OAuth 授权码流程，应作为独立迁移，不得让 KVideo 服务端获得用户的 SyncHub 密码。

## 6. 分阶段路线

### 阶段 0：契约与文档基线

目标：消除产品歧义，冻结首版接口语义。

SyncHub 交付：

- 定义套餐、订阅状态、权益、设备、同步版本和冲突的 API 契约。
- 明确 Free 账户是否允许创建仅用于试用的 Key；MVP 默认不允许读写付费 collection。
- 错误码区分未认证、订阅无效、权益不足、设备超限、版本冲突和配额超限。

KVideo 交付：

- 所有公开文档声明仅售卖同步服务。
- 界面文案不把 `/premium`、来源或播放能力描述为 VIP 权益。
- 建立契约类型和模拟响应测试，但不改变现有同步行为。

验收：两个仓库的契约示例一致；Markdown 链接有效；现有测试和构建通过。

### 阶段 1：Pro 权益与手动开通

目标：不接支付，先证明服务端权益闭环。

SyncHub 交付：

- 将套餐到权益和配额的映射集中到一个领域服务。
- 提供当前订阅、权益和限制的只读接口。
- 提供受审计的管理员手动开通、续期、到期和取消能力。
- Free、过期、取消和 Pro 状态建立完整测试矩阵。

KVideo 交付：

- 连接 SyncHub 后读取并显示当前套餐、权益、到期时间。
- 订阅不可用时停止云端请求并保持本地读写正常。
- 设置页显示可操作错误，不用 `console.error` 代替用户反馈。

验收：管理员可以给测试账户开通 Pro；KVideo 无需刷新即可识别权益；Free 和过期账户不能写入付费同步数据；本地数据不丢失。

### 阶段 2：设备管理与密钥治理

目标：建立最多 5 台设备的可靠限制和撤销路径。

SyncHub 交付：

- KVideo 设备登记、列表、重命名、心跳和撤销 API。
- Key 与设备建立可审计关联，撤销设备后对应凭据立即失效。
- 设备上限在事务内校验，避免并发注册越限。

KVideo 交付：

- 首次启用同步时生成稳定设备 ID 并登记设备。
- 展示当前设备、其他设备、最后同步时间和撤销操作。
- 被撤销设备进入明确的“需要重新连接”状态。

验收：第 6 台设备注册被稳定拒绝；撤销后不能继续同步；重新授权不影响本地数据；并发和越权测试通过。

### 阶段 3：无损多设备同步协议

目标：替换整份数组最后写入覆盖。

SyncHub 交付：

- 支持基础版本或 ETag 的条件写入，冲突返回专用错误。
- 支持逐记录变更或等价的可合并协议，并保留删除标记。
- 所有写入具备幂等请求 ID，网络重试不会重复应用。

KVideo 交付：

- 本地待同步队列、指数退避、在线恢复和并发请求合并。
- 历史、收藏的确定性合并规则及删除同步。
- 显示同步中、已同步、待同步、冲突、需要登录和服务不可用状态。
- 从旧整文档格式无损迁移，并保留回滚读取能力一个发布周期。

验收：两台离线设备交叉增加、修改和删除数据后不会静默丢失；断网重试不产生重复记录；旧数据能迁移；故障注入测试通过。

### 阶段 4：设置和敏感来源配置同步

目标：同步用户偏好，同时保护用户自行配置的来源秘密。

SyncHub 交付：

- 增加 `settings` 和 `source-config` collection 及独立大小限制。
- 服务端只验证密文封装结构，不记录或解析来源正文。
- 增加密钥版本、加密算法版本和恢复所需元数据字段。

KVideo 交付：

- 明确哪些设置可跨设备，排除会话、缓存、设备专属设置和运行时 URL。
- 使用 Web Crypto 对来源配置加密后上传；解密密钥不发送给 SyncHub。
- 提供恢复密钥或恢复短语流程，并明确遗失密钥无法由服务端找回。
- 导入前预览差异，冲突时让用户选择本地、云端或另存备份。

验收：数据库、日志和管理端均看不到来源地址与请求头明文；错误密钥不能破坏云端旧版本；跨浏览器恢复演练通过。

### 阶段 5：备份、版本历史与恢复

目标：形成用户可感知的 Pro 服务价值。

SyncHub 交付：

- 为 KVideo collection 建立不可变快照和默认 30 天保留策略。
- 提供快照列表、恢复预览、恢复和删除接口。
- 恢复动作创建新版本而不是覆盖历史，并写入审计事件。

KVideo 交付：

- 提供“立即备份”、版本列表、变更摘要和恢复确认。
- 恢复前自动保存当前状态，避免误操作造成二次丢失。

验收：历史、收藏、设置和加密来源配置均可恢复；超出保留期自动清理；恢复失败保持原数据不变。

### 阶段 6：支付与自动续费

目标：在同步服务稳定后接入真实商业闭环。

SyncHub 交付：

- 选择支付渠道后实现结账、Webhook、续费、取消、退款和宽限期。
- Webhook 以提供商事件 ID 幂等处理，金额、币种和套餐均由服务端配置决定。
- 支付成功页只展示结果，不能直接修改订阅状态。
- 保留账单事件、失败原因和人工修复审计记录。

KVideo 交付：

- 展示套餐、订阅状态、到期时间和管理订阅入口。
- 支付与账单操作跳转 SyncHub；KVideo 不保存支付凭据。

验收：测试环境覆盖首次购买、重复 Webhook、续费失败、取消续订、到期、退款和恢复订阅；对账结果一致后才能上线。

## 7. 兼容迁移

### 7.1 `premium` 命名迁移

1. 文档先统一称为“独立数据空间”，保留路径和环境变量的兼容说明。
2. 代码增加中性的新存储键和内部类型，并能读取旧 `premium` 键。
3. 一个发布周期内双读、只写新格式，提供迁移测试。
4. 后续再决定新路由名称；旧 `/premium` 使用重定向保留兼容。
5. `PREMIUM_PASSWORD` 只有在新变量稳定后才弃用，不能静默失效。

这项迁移与 SyncHub Pro 套餐没有依赖关系，应单独开发和提交。

### 7.2 旧同步路径

- Redis 同步是 KVideo 自托管兼容能力，不自动获得 SyncHub 的设备、备份和版本权益。
- SyncHub 新协议上线时，旧 `watch-history` 和 `favorites` 文档至少保留一个版本周期的读取迁移。
- 任何迁移都先复制再切换，不在首次成功读取前删除旧数据。

## 8. 测试与发布门禁

每个小功能只在相匹配的验证通过后提交；跨仓库功能分别提交，不制造一个同时修改两个仓库的“原子提交”假象。

KVideo 常规门禁：

```powershell
pnpm lint
pnpm test
pnpm test:e2e
pnpm build
pnpm cf:build
git diff --check
```

涉及 Docker 时额外运行：

```powershell
docker compose config
docker build -t kvideo .
```

SyncHub 常规门禁：

```powershell
gofmt -w <changed-go-files>
go test ./...
go vet ./...
git diff --check
```

涉及管理端和发布链路时额外运行仓库现有的 `scripts/test-mvp.ps1`、Docker smoke 和发布验证脚本。

每阶段至少覆盖：领域单元测试、仓储集成测试、API 权限测试、KVideo 客户端状态测试和一个真实浏览器端到端流程。支付阶段必须增加 Webhook 重放与乱序测试。

## 9. 建议提交顺序

以下每项是一个可独立验证的功能边界，不要求机械地一天完成一项：

1. `docs: define paid sync service boundary`
2. `feat(subscription): expose effective sync entitlements`（SyncHub）
3. `feat(sync): display subscription entitlement status`（KVideo）
4. `feat(devices): register KVideo sync devices`（SyncHub）
5. `feat(sync): manage registered devices`（KVideo）
6. `feat(metadata): reject stale document versions`（SyncHub）
7. `feat(sync): merge history changes without data loss`（KVideo）
8. `feat(sync): preserve favorite deletions across devices`（KVideo）
9. `feat(metadata): store encrypted KVideo source configuration`（SyncHub）
10. `feat(sync): encrypt source configuration before upload`（KVideo）
11. `feat(backup): retain KVideo metadata snapshots`（SyncHub）
12. `feat(sync): restore a cloud snapshot`（KVideo）

支付接入必须排在上述同步、设备和恢复能力稳定之后。

## 10. MVP 完成定义

只有同时满足以下条件，付费同步 MVP 才算完成：

- Pro 用户最多在 5 台登记设备间同步观看历史和收藏。
- 多设备离线修改、删除和网络重试不会造成静默数据丢失。
- Free、过期、撤销和设备超限状态均由 SyncHub 服务端可靠拒绝。
- KVideo 在任何云端失败状态下仍可正常使用本地功能。
- 用户能看到同步状态、最后成功时间、失败原因和恢复动作。
- 用户能查看并撤销设备，撤销立即生效。
- 文档、界面和套餐页均不暗示平台销售或提供视频资源。
- 两个仓库各自通过对应测试、构建和发布门禁。

## 11. 暂不实施

- 按视频源、内容、清晰度或播放线路划分套餐。
- 平台来源市场、付费来源目录或来源推荐服务。
- 家庭组、团队空间、内容分享和社交功能。
- 在稳定性与成本数据不足时承诺无限设备、无限历史或永久备份。
- 在同步协议、恢复能力和订阅状态机验证前直接接入生产支付。
