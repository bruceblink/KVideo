# KVideo 播放器优先开发计划

## 1. 战略结论

KVideo 当前首先要证明的不是同步服务能否收费，而是用户能否稳定、顺畅地完成一次播放。同步服务依赖持续使用、多设备需求和数据沉淀；播放器没有形成留存之前，提前建设计费只会增加维护成本。

当前主线按以下顺序执行：

```text
播放可靠性 -> 故障恢复 -> 跨端体验 -> 质量度量 -> 用户留存 -> 同步服务商业化
```

同步相关工作暂时限制在安全漏洞、数据丢失和严重兼容问题。套餐、设备收费、版本历史、支付和自动续费均不进入当前迭代。

## 2. 产品边界

1. KVideo 负责播放用户自行合法配置的来源，不提供或销售视频资源。
2. 播放成功不能只用“页面打开”衡量，必须完成详情加载、媒体加载、首帧出现和持续播放。
3. 外部来源故障与 KVideo 自身故障需要分开统计，避免用来源不可用掩盖客户端缺陷。
4. 自动恢复不得无限循环、悄悄跳集或丢失播放进度。
5. 新播放器功能必须覆盖桌面和移动端；涉及全屏、PiP 或原生 HLS 时必须单独验证对应平台。
6. 不以大规模视觉重构替代可靠性建设。界面调整服务于状态可见、操作清晰和触控可用。

## 3. 当前基线

### 3.1 已有能力

- HLS.js 与浏览器原生 HLS 两条播放路径。
- 直连、始终代理和失败后代理重试策略。
- 手动重试、备选来源发现与切源时保留播放位置。
- 断点续播、自动连播、片头片尾跳过、倍速和音量控制。
- 系统全屏、网页全屏、画中画、Cast 和 Android WebView 兼容逻辑。
- 清晰度探测、卡顿检测、弹幕和广告过滤回退。

### 3.2 主要风险

- 播放恢复状态分散在页面、播放器组件和多个 Hook 中，缺少统一状态与确定的恢复顺序。
- 手动重试最多 20 次且直连/代理交替，用户难以知道每次尝试发生了什么。
- 卡顿检测阈值为 200ms，容易把正常缓冲或调度抖动误判为卡顿。
- HLS 错误分类、恢复和销毁逻辑集中在较大的 Hook 中，React Compiler 当前会跳过其中的内联类。
- 换源、换集、自动连播和离开页面分别保存进度，存在竞态和旧状态写回风险。
- 自动化测试偏向纯函数；尚无使用受控媒体样本的真实播放端到端基线。
- 缺少不采集标题、来源 URL 和请求头的隐私友好播放质量指标。

## 4. 核心指标

### 4.1 事件定义

| 指标 | 定义 |
| --- | --- |
| 播放尝试 | 用户选择某集并产生一次新的播放会话 ID |
| 启动成功 | 媒体进入可播放状态并实际播放至少 5 秒 |
| 首帧时间 | 从用户确认播放到首次 `playing` 事件的时间 |
| 有效卡顿 | 首帧后非用户暂停、拖动或切集导致的连续缓冲超过 2 秒 |
| 自动恢复成功 | 发生可恢复故障后，不需要用户操作并在 10 秒内恢复播放 |
| 进度正确 | 退出、刷新、换源或换集后恢复位置误差不超过 10 秒 |
| 播放会话崩溃 | 页面异常终止、未处理错误或播放器进入无法操作状态 |

来源返回 404、空播放列表或不受浏览器支持的编码应标记为来源或能力限制；状态机死锁、错误重试、进度丢失和 UI 无法操作属于 KVideo 缺陷。

### 4.2 隐私规则

- 默认不发送视频标题、媒体 URL、来源地址、请求头、搜索词、账户名或 IP 标签。
- 只记录随机播放会话 ID、浏览器能力类别、播放路径、阶段耗时、错误分类和恢复结果。
- 自托管实例的远程质量上报默认关闭；本地诊断信息可由部署者主动导出。
- 在实现正式遥测前，先通过受控媒体样本和人工测试矩阵建立基线。

## 5. 分阶段路线

### P0：受控播放测试基线

目标：让播放器修改可以重复验证，而不是依赖随时变化的外部来源。

交付：

- 在测试环境提供短 VOD HLS、主/子播放列表、损坏清单、超时、404 和中途断流样本。
- 为媒体请求建立 Playwright 路由或本地测试服务器，不提交有版权风险的内容。
- 覆盖直连成功、代理回退、原生 HLS 能力分支和不可恢复错误。
- 建立桌面 Chromium 的首条真实播放冒烟流程，验证播放、暂停、拖动和进度保存。
- 修复播放器相关 lint 警告和 React Compiler 跳过，保持行为不变。

验收：测试不访问第三方媒体服务；失败可稳定复现；单元测试、播放 E2E、lint 和生产构建通过。

### P1：统一播放状态与错误模型

目标：所有恢复行为由一个明确的状态机驱动。

建议状态：

```text
idle -> loading_detail -> loading_media -> playing
                       -> recovering -> playing
                       -> blocked | failed
playing -> buffering -> recovering | playing | failed
```

交付：

- 定义 `PlaybackErrorKind`，至少区分详情失败、清单网络失败、媒体网络失败、格式/编码不支持、代理不可用、超时和未知错误。
- 将播放路径、尝试次数、恢复动作和最终错误集中到可测试的状态转换模块。
- 每个播放会话和媒体切换使用新的 generation ID，旧异步结果不得覆盖新状态。
- 将 20 次交替重试改为有上限、有说明的恢复策略。
- 错误界面展示当前失败类别以及“重试当前路径、尝试另一线路、返回来源列表”等真实动作。

验收：所有状态转换和恢复上限有单元测试；错误不会形成循环；切集或切源后旧错误不会重新出现。

### P2：自动恢复与线路切换

目标：常见网络和来源故障尽可能在一次用户操作内恢复。

恢复顺序由设置和运行环境决定，默认候选为：

```text
当前路径轻量恢复 -> 重载当前媒体 -> 直连/代理切换 -> 同内容备选来源 -> 明确失败
```

交付：

- 使用浏览器和 HLS.js 事件判断真实卡顿，连续缓冲达到合理阈值后才进入恢复。
- 区分清单故障、分片故障和解码故障，不对不可恢复编码错误重复网络请求。
- 换线路前保存当前位置、暂停状态、音量、倍速和全屏意图。
- 自动换源仅选择同一标题和目标集数可确认匹配的候选；无法确认时交给用户选择。
- 为每个恢复动作设置次数和总时间预算，并允许用户取消。

验收：断流、超时和代理失败场景能按确定顺序恢复；恢复后进度误差不超过 10 秒；无无限请求或线路抖动。

### P3：进度、选集与连续播放一致性

目标：任何导航和恢复操作都不丢进度、不跳错集。

交付：

- 建立稳定的内容与剧集身份规则，避免只依赖标题字符串。
- 统一进度写入入口，处理定时保存、暂停、页面隐藏、卸载、换源和换集。
- 使用单调时间和 generation ID 防止旧播放会话覆盖新进度。
- 明确接近结尾时从头播放还是标记完成，并允许用户选择。
- 自动下一集显示可取消倒计时；倒序列表和最后一集行为一致。
- 对来源之间集数命名不一致提供匹配失败提示，不静默播放错误集数。

验收：刷新、后退、换源、换集、自动连播和浏览器崩溃恢复场景有测试；进度误差满足指标；不会跨集写错历史。

### P4：移动端、全屏与画中画

目标：在真实设备上完成主流播放操作，不依赖桌面布局缩放。

交付：

- 建立 Android Chrome/WebView、iPhone Safari 和桌面 Chrome/Edge/Safari 支持矩阵。
- 验证触摸进度、双击快进、控制条显隐、方向切换、安全区域和系统返回键。
- 统一网页全屏、系统全屏和 PiP 的进入、退出及状态恢复规则。
- 自动播放受阻时提供明确的点击播放状态，不将其显示为媒体故障。
- 键盘、遥控器和屏幕阅读器可操作核心控制，焦点不会困在菜单中。

验收：支持矩阵中的必测场景有记录；无控制遮挡、无法退出全屏或 PiP 后状态错乱等阻断问题。

### P5：性能与长时间稳定性

目标：播放器可持续使用，不因长视频、多次切换或后台恢复逐渐失效。

交付：

- 测量首帧、内存、主线程长任务、HLS 实例和 Blob URL 生命周期。
- 清理重复定时器、事件监听器、媒体请求和过期清晰度探测。
- 对大集数列表和多来源列表做虚拟化或等价的渲染控制，但只在测量证明需要时实施。
- 运行至少 2 小时播放、20 次换集、20 次换源和反复前后台切换的稳定性测试。
- 将播放器诊断信息整理为可导出的本地报告，避免泄露敏感来源配置。

验收：压力场景没有持续内存增长、重复请求风暴或失效控制；性能基线进入发布检查。

### P6：用户反馈与增长验证

目标：证明播放体验带来了持续使用和多设备需求。

交付：

- 增加播放器内问题分类反馈，但不自动附带标题或来源秘密。
- 建立版本级播放质量看板或本地汇总流程。
- 每个版本先解决成功率和恢复率回退，再增加次要播放功能。
- 通过公开路线和更新日志说明修复了哪些真实播放问题。

验收：连续 4 周有稳定质量数据；高频失败类别有负责人和修复闭环；不存在持续增长的 P0/P1 播放阻断问题。

## 6. 同步商业化启动门槛

以下三类条件必须同时满足，才重新评审同步服务商业化。数值是首轮决策门槛，可根据实际部署规模调整，但调整必须记录原因。

### 6.1 播放质量门槛

- 受控样本启动成功率至少 99%。
- 可归因于 KVideo 的真实播放启动成功率至少 95%，连续 4 周无显著回退。
- P75 首帧时间不高于 3 秒；统计时单独展示外部来源响应耗时。
- 可恢复网络故障的自动恢复成功率至少 80%。
- 崩溃或不可操作会话低于 0.5%。
- 支持矩阵内没有未解决的 P0 播放问题。

### 6.2 用户需求门槛

满足下面任一组，并确认数据来自持续活跃而非一次性访问：

- 至少 500 周活跃播放用户，连续 4 周保持；或
- 至少 100 名用户明确存在两台及以上设备的使用需求，并有可验证的同步候补名单。

同时需要观察播放用户的 4 周留存趋势。如果新增用户上涨但留存持续下降，应继续修播放器而不是启动收费。

### 6.3 商业可行性门槛

- 完成 SyncHub 单活跃用户的存储、请求、带宽、支付和支持成本测算。
- 保守付费转化假设下，目标套餐毛利率至少 70%。
- 至少 5% 的活跃用户明确表达对跨设备同步或恢复能力的付费意愿。
- 同步 MVP 的安全、冲突和恢复设计已经通过评审，不以用户数据承担试错成本。

达到门槛不代表立即收费，只代表可以恢复 [同步服务开发计划](sync-service-development-plan.md) 的阶段 1，并重新核对套餐与价格。

## 7. 缺陷优先级

| 级别 | 定义 | 处理规则 |
| --- | --- | --- |
| P0 | 大范围无法播放、数据损坏、安全问题或无法退出的播放器状态 | 停止其他开发，立即修复并发布 |
| P1 | 主流环境稳定复现的播放失败、严重卡顿、错集或进度丢失 | 当前版本优先，不带到下个功能版本 |
| P2 | 有替代路径的交互缺陷、特定设备兼容问题或明显性能退化 | 排入最近里程碑并增加回归测试 |
| P3 | 视觉细节、低频便利性和非阻断增强 | 不抢占可靠性工作 |

## 8. 测试与提交规则

每个功能完成后先运行与风险匹配的验证，通过后单独提交。不能测试、测试失败或只完成一部分时，不提交为完成状态。

基础门禁：

```powershell
pnpm lint
pnpm test
pnpm test:e2e
pnpm build
git diff --check
```

涉及运行平台时增加对应的 Cloudflare、Docker 和真实设备验证。播放器状态、错误分类、进度规则等非平凡函数需添加简短函数级注释，说明输入、输出和关键副作用。

## 9. 建议提交顺序

1. `test(player): add controlled HLS playback fixtures`
2. `test(player): cover direct playback and proxy recovery`
3. `fix(player): remove compiler and hook warnings`
4. `feat(player): classify playback failures`
5. `feat(player): centralize recovery transitions`
6. `fix(player): ignore stale media session results`
7. `feat(player): recover stalled streams within a budget`
8. `feat(player): preserve state across source switches`
9. `fix(player): serialize playback progress updates`
10. `feat(player): make auto-next cancellable`
11. `test(player): cover mobile fullscreen and PiP transitions`
12. `perf(player): verify long-running playback stability`

## 10. 当前完成定义

“播放器已经能够立足”至少意味着：

- 有不依赖第三方来源的真实播放自动化测试。
- 失败类别、恢复路径和最终状态对用户可见且可预测。
- 断流、超时、代理失败、换源和换集不会造成无限重试或进度丢失。
- 桌面与移动端支持矩阵完成真实设备验证。
- 连续 4 周满足播放质量门槛，并形成稳定的用户反馈闭环。
- 达到用户需求门槛后，团队才重新评审同步服务商业化。
